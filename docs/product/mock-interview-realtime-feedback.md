# 模拟面试实时反馈方案

## 1. 目标

模拟面试保持 10 轮固定结构，但每一轮都必须基于历史回答、候选人画像、岗位要求和当前面试状态重新生成下一问。

成功标准：

- 用户感觉自己在被真实面试官连续追问，而不是回答固定题库
- 每轮首 token 体感快，目标是 3 秒内看到反馈或问题开始输出
- 每轮反馈具体、克制、可执行，保持 2-3 句
- 10 轮结束后能沉淀到简历优化，而不是只生成一份面试报告

硬约束：

- 实时反馈禁止使用 mock 兜底
- 下一轮面试问题禁止使用 mock 兜底
- 失败时只能使用真实重试、排队、恢复、错误提示或稍后继续
- 不允许为了速度返回固定题库、占位问题或模板化假反馈

## 2. 10 轮结构

Phase 1 固定为：

| 轮次 | 面试官 | 主要考察 |
| --- | --- | --- |
| 1 | Hiring Manager | 项目背景与岗位动机 |
| 2 | Hiring Manager | 产品问题定义 |
| 3 | Hiring Manager | AI / LLM 应用理解 |
| 4 | Hiring Manager | 数据分析与指标判断 |
| 5 | Hiring Manager | 用户研究与需求洞察 |
| 6 | Hiring Manager | 追问、取舍、失败复盘 |
| 7 | HR | 求职动机与稳定性 |
| 8 | HR | 公司与行业理解 |
| 9 | HR | 协作、压力与沟通 |
| 10 | HR | 总结表达与反问准备 |

轮次结构固定，问题内容动态生成。

## 3. 每轮上下文装配

每一轮生成前拉取四类上下文。

### 3.1 候选人背景包

- 简历解析结果
- 咨询阶段候选人画像
- AI 产品经理五维能力标签
- 用户目标城市、公司偏好、项目和实习经历

### 3.2 岗位上下文包

- 当前岗位线索
- 公司方向
- 结构化岗位要求
- 相关面经和知识库证据
- JD gap map

### 3.3 面试状态包

- 当前轮次
- 当前面试官身份
- 已覆盖能力点
- 未覆盖能力点
- 已发现风险点
- 当前回答质量趋势
- 是否需要追问、换题、加压或收束

### 3.4 历史记忆包

- 最近 2-3 轮保留原文
- 更早轮次压缩为结构化摘要
- 全量原始历史存库，用于审计、回放和后续简历优化，不默认全部塞入 prompt

## 4. 每轮运行流程

```text
用户提交回答
  -> 保存原始回答
  -> 评估本轮回答
  -> 更新 interview_state
  -> 更新 candidate_signal_delta
  -> 生成 2-3 句实时反馈
  -> 判断下一轮策略
  -> 生成下一轮问题
  -> 流式返回
```

推荐返回顺序：

1. 先返回真实链路的处理状态
2. 再返回实时反馈
3. 再流式返回下一轮问题

禁止 mock 兜底后，速度优化不能来自“先塞一个假结果”，只能来自：

- 更小的 prompt payload
- 更快的状态读取
- 更稳定的流式输出
- 更明确的重试边界
- 更少但更准的模型调用

## 5. 实时反馈功能槽

实时反馈不是总结报告，也不是老师批作业。它应该承担三个职责：

1. 告诉用户刚才这轮答得怎样
2. 点出一个最关键的提升方向
3. 为下一问制造自然过渡

默认结构：

```text
判断句：这轮回答的主要质量判断。
改进句：一个具体可执行的改进点。
过渡句：自然接到下一问，必要时可省略。
```

示例：

```text
你这轮把项目背景讲清楚了，但产品目标还停留在功能层。
真实面试里建议补一句：这个功能解决了谁的什么问题，以及你怎么判断它值得做。
我会继续追问你当时的指标设计。
```

## 6. 反馈风格设置

建议提供 3 个反馈风格，但 Phase 1 默认只展示一个主模式，其他作为内部可配置项。

### 6.1 默认：真实面试官模式

适合主流程默认体验。

特点：

- 克制
- 直接
- 不过度安慰
- 不使用教学腔
- 每轮只抓一个最重要问题

语气标准：

- 可以说“这个回答还不够面试化”
- 可以说“这里缺少证据”
- 避免说“你太棒了”“非常优秀”这类空泛鼓励
- 避免把反馈写成完整教程

### 6.2 辅导模式

适合用户主动要求“教我怎么改”。

特点：

- 更解释性
- 给出可套用表达
- 允许多一点鼓励
- 仍然不能超过 3 句

适用触发：

- 用户连续两轮低质量回答
- 用户点击“给我更具体建议”
- 面试结束后的复盘页

### 6.3 压力面模式

适合后续增强，不建议 Phase 1 默认打开。

特点：

- 更像大厂压力追问
- 反馈更短
- 下一问更尖锐
- 用于验证回答真实性和边界

使用限制：

- 不对低信心用户默认启用
- 不做羞辱式评价
- 不制造人格攻击

## 7. 专业度设置

专业度不建议让用户自由调太多档。Phase 1 推荐内部三档，默认中高专业度。

### 7.1 L1：入门解释型

适合完全不了解 AI 产品经理面试的用户。

反馈特征：

- 解释术语
- 多给表达模板
- 少用面试官压迫感

示例：

```text
你提到了用户痛点，但还没有说明怎么验证它是真的。
面试里可以补充调研样本、用户反馈来源或数据指标，这会让判断更可信。
```

### 7.2 L2：中高专业度，默认

适合 Phase 1 主体验。

反馈特征：

- 使用真实产品面试语言
- 关注问题定义、证据、指标、取舍
- 不解释常见概念
- 每轮只给一个关键改进点

示例：

```text
这个回答有场景，但缺少指标闭环。
建议补出你当时如何定义成功，以及上线后用什么数据判断方向对不对。
```

### 7.3 L3：强专业面试官

适合用户已准备较充分，或进入压力面模式。

反馈特征：

- 更关注逻辑漏洞
- 更少安慰
- 追问会要求量化和反事实思考

示例：

```text
这个回答的问题是结论先行，证据不足。
如果你无法说明为什么这个方案优先级高于其他方案，面试官会继续追问决策依据。
```

## 8. 推荐默认配置

Phase 1 建议默认：

```json
{
  "feedbackStyle": "real_interviewer",
  "professionalLevel": "L2",
  "feedbackLength": "2_to_3_short_sentences",
  "questionStrategy": "adaptive_follow_up",
  "tone": "direct_but_constructive",
  "allowPressure": true,
  "pressureDefault": false,
  "mockFallback": false
}
```

这个默认值的产品判断是：

- L1 太像课程，不像面试
- L3 对应届生太硬，容易劝退
- L2 最像“真实但仍然有帮助”的作品集级体验

## 9. 下一问策略

每轮生成下一问前，先做策略判断。

| 策略 | 使用条件 | 下一问方向 |
| --- | --- | --- |
| follow_up | 用户回答有价值但不够深 | 沿当前话题追问 |
| evidence_check | 回答空泛或像包装话术 | 要求举例、数据或过程 |
| dimension_shift | 当前维度已经覆盖 | 切到下一能力维度 |
| pressure_probe | 发现逻辑漏洞 | 用取舍、反例或失败复盘加压 |
| rescue | 用户明显答非所问 | 缩小问题范围，拉回主题 |

内部输出结构建议：

```json
{
  "turn": 4,
  "interviewer": "hiring_manager",
  "feedback": {
    "style": "real_interviewer",
    "professionalLevel": "L2",
    "sentences": []
  },
  "stateUpdate": {
    "coveredDimensions": [],
    "newSignals": [],
    "risks": [],
    "nextStrategy": "follow_up"
  },
  "nextQuestion": {
    "targetDimension": "data_analysis",
    "questionType": "follow_up",
    "question": "",
    "reason": "",
    "expectedSignal": []
  }
}
```

## 10. 禁止项

实时反馈不能：

- 每轮都夸用户
- 输出长篇教学
- 泛泛说“表达要更清楚”
- 引入用户没有说过的经历
- 直接替用户编造成果
- 变成最终总结报告
- 用“已验证真实岗位”之类过度承诺话术
- 使用 mock、固定题库或占位内容伪装成真实生成结果

## 11. 质量与速度权衡

禁止 mock 兜底后，优先级应调整为：

```text
状态正确性 > 问题质量 > 首 token 速度 > 完整输出速度 > 视觉流畅感
```

原因：

- 模拟面试的核心价值是真实连续追问
- 一旦用户发现问题是假的或不记得上下文，信任会直接崩
- 可以接受慢 1-2 秒，但不能接受问出和上一轮无关的问题

### 11.1 可接受的速度策略

可以做：

- 先流式展示“正在分析你的回答”，但这只能是 UI 状态，不是 AI 反馈
- 将状态更新和下一问生成合并为一次强模型调用
- 最近 2-3 轮保留原文，更早历史使用结构化摘要
- 提前缓存岗位上下文、面经证据和 JD gap map
- 在用户作答时预加载静态上下文
- 对模型输出做结构化校验，不合格则真实重试一次

不可以做：

- 生成失败时返回预置反馈
- 模型超时后切固定题库
- 用模板问题假装动态追问
- 用“你回答得不错”这类泛化句子冒充实时反馈

### 11.2 推荐调用策略

每轮默认使用一次主调用：

```text
输入：候选人背景包 + 岗位上下文包 + 面试状态包 + 历史记忆包 + 用户本轮回答
输出：本轮反馈 + 状态更新 + 下一轮问题 + 质量自检字段
```

这样比拆成“反馈调用 + 问题调用”更稳：

- 少一次网络和模型排队延迟
- 反馈和下一问共享同一套判断
- 更容易保证追问连续性

仅在主调用失败或结构化校验不通过时，允许真实重试一次。重试仍然必须走同一真实上下文，不允许降级到 mock。

### 11.3 失败处理

如果真实生成失败：

```text
第一次失败：自动真实重试一次
第二次失败：提示用户当前轮生成失败，允许重新生成或稍后继续
断线失败：保留当前轮状态，恢复后继续
结构化校验失败：不展示坏输出，真实重试
```

用户可见提示建议：

```text
这一轮问题生成失败了，我没有用固定题库替你糊过去。你可以重新生成这一轮，或稍后从当前轮继续。
```

这句话略硬，但符合产品信任原则：真实比假顺滑更重要。

## 12. 验收指标

产品验收：

- 至少 80% 抽样轮次能正确引用前文或候选人背景
- 至少 80% 抽样反馈包含具体改进点，而不是泛泛评价
- 至少 80% 抽样下一问与当前岗位或候选人回答有关
- 每轮反馈保持 2-3 句
- 0 次 mock、固定题库或占位内容进入用户可见面试流

技术验收：

- 每轮首 token 目标低于 3 秒
- 如果强模型真实生成链路超过 3 秒，允许先展示非 AI 的处理状态
- 10 轮状态机可以完整结束
- 第 6 轮后能稳定切换到 HR 视角
- 断线重连后可以恢复历史和当前轮次
- 结构化输出校验失败时，不展示坏输出

人工评审问题：

- 这像不像真实 AI 产品经理面试？
- 它有没有记住用户刚才说过什么？
- 反馈有没有指出一个可执行的改进？
- 下一问是不是自然，而不是随机？
- 有没有任何内容看起来像模板兜底？

## 13. 当前推荐结论

默认实时反馈应采用“真实面试官模式 + L2 中高专业度”。

这能同时满足三件事：

1. 对用户有训练价值
2. 对作品集展示有专业感
3. 不会因为过度压力或过度教学破坏连续面试体验

在禁止 mock 兜底的前提下，推荐进一步采用：

- 单次主模型调用生成“反馈 + 状态更新 + 下一问”
- UI 先展示真实处理状态，而不是假反馈
- 输出失败时真实重试一次，仍失败则诚实中断并允许恢复
- 质量优先于极限速度，首 token 目标从“必须 3 秒内 AI 内容”调整为“3 秒内有真实处理进度，AI 内容尽快流式输出”
